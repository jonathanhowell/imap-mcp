# Phase 11.1: unflag_message tool - Context

**Gathered:** 2026-03-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Add an `unflag_message` MCP tool that removes a custom IMAP keyword from a message using IMAP STORE `-FLAGS`. This is the direct counterpart to `flag_message` (Phase 11). No new capabilities beyond keyword removal — bulk removal and other flag operations belong in separate phases.

</domain>

<decisions>
## Implementation Decisions

### Poller cache update
- Yes — add a `removeKeyword(accountId, uid, keyword)` method to the Poller class, mirroring `updateKeyword`
- Use case-insensitive comparison (`.toLowerCase()`) when finding and removing the keyword — consistent with Phase 11 decision and IMAP server normalization behavior
- Call `poller.removeKeyword()` after a successful `messageFlagsRemove` call, same pattern as `flag_message` calling `poller.updateKeyword()`

### "Keyword not found" behavior
- Always return `{ success: true, account, uid, keyword }` — idempotent, consistent with IMAP protocol semantics
- Do NOT add a `was_present` field — that would require an extra IMAP round-trip and is not needed
- Mirrors `flag_message` which doesn't check for duplicate adds before calling `messageFlagsAdd`

### Structural patterns (carry-forward from Phase 11)
- `messageFlagsRemove([uid], [keyword], { uid: true })` — single UID wrapped in array, same signature pattern as `messageFlagsAdd`
- PERMANENTFLAGS check: include the same `client.mailbox` guard with warning-only log (for symmetry and to catch servers that don't support custom keywords)
- Tool schema: same required fields as `flag_message` — `account`, `uid`, `keyword`; optional `folder` (default: INBOX)

### Claude's Discretion
- Exact JSDoc/comment wording
- Test file structure (follow `flag-message.test.ts` as template)

</decisions>
