## Summary

Phase 11.1 adds an `unflag_message` MCP tool as the direct counterpart to `flag_message` (Phase 11). The implementation is a near-identical mirror of `flag_message` — three files change: a new `src/tools/unflag-message.ts`, an addition to `src/polling/poller.ts`, and registration in `src/index.ts`. No new patterns are introduced; every pattern comes directly from the Phase 11 implementation.

The canonical template is `src/tools/flag-message.ts`. The only behavioral changes are: (1) `messageFlagsAdd` becomes `messageFlagsRemove`, (2) the Poller gets a `removeKeyword` method that filters out rather than appends, (3) the tool name, description, and log messages are updated to reflect removal semantics. Everything else — error handling, lock pattern, PERMANENTFLAGS guard, return shape — is identical.

**Primary recommendation:** Copy `flag-message.ts` verbatim, make the four targeted substitutions documented in Canonical References, add `removeKeyword` to Poller adjacent to `updateKeyword`, then register in `src/index.ts`.

### Pattern 2: Poller.removeKeyword — Filter Instead of Append
**What:** `removeKeyword` mirrors `updateKeyword` but filters the keyword out of the array instead of appending it.
**When to use:** Must be placed immediately after `updateKeyword` at line ~157 in poller.ts.
**Example (derived from `src/polling/poller.ts` lines 143-156):**

```typescript
// Source: src/polling/poller.ts updateKeyword() (template — filter instead of append)
removeKeyword(accountId: string, uid: number, keyword: string): void {
  const entries = this.cache.get(accountId);
  if (!entries) return;
  const msg = entries.find((m) => m.uid === uid);
  if (!msg) return;
  msg.keywords = (msg.keywords ?? []).filter(
    (k) => k.toLowerCase() !== keyword.toLowerCase()
  );
}
```

### Anti-Patterns to Avoid
- **Checking whether keyword is present before removing:** CONTEXT.md explicitly forbids this — idempotent behavior is correct and avoids an extra IMAP round-trip.
- **Using `messageFlagsSet` instead of `messageFlagsRemove`:** `messageFlagsSet` replaces ALL flags; `messageFlagsRemove` is STORE `-FLAGS` (subtractive).
- **Returning `was_present` field:** Not wanted — locked decision.
- **Returning error when keyword is absent:** IMAP semantics make this a no-op; return success.

## Common Pitfalls

### Pitfall 1: Forgetting `{ uid: true }` in messageFlagsRemove
**What goes wrong:** Without `{ uid: true }`, imapflow interprets the first argument as sequence numbers, not UIDs — wrong message is modified or no message found.
**Why it happens:** imapflow API supports both UID mode and sequence-number mode; UID mode requires explicit opt-in.
**How to avoid:** Copy the exact call signature from `messageFlagsAdd` and replace method name only.
**Warning signs:** STORE applies to unexpected message, or returns successfully but no flag removed.

### Pitfall 2: Wrapping uid in array
**What goes wrong:** Passing `uid` (number) instead of `[uid]` (array) causes a type error or runtime failure.
**Why it happens:** imapflow `messageFlagsRemove` first arg is `SequenceString | number[]`.
**How to avoid:** Always `[uid]` — single-element array. Identical to `messageFlagsAdd` call in flag-message.ts.

### Pitfall 3: Not handling the "message not in cache" case in removeKeyword
**What goes wrong:** If the message isn't in the poller cache (e.g., old message), `msg` is undefined. Accessing `msg.keywords` throws.
**Why it happens:** Cache only holds recent messages (last 30 days seed, incremental after).
**How to avoid:** Guard with `if (!msg) return;` — same pattern as `updateKeyword`.

### Pitfall 4: lock.release() outside finally block
**What goes wrong:** If `messageFlagsRemove` throws, the mailbox lock is never released, blocking all subsequent IMAP operations on that folder.
**Why it happens:** Error path skips release call.
**How to avoid:** Always `lock.release()` in `finally` — copy the try/finally structure exactly from `flag-message.ts`.
